# Muibook Design

Muibook documents the Mui design system as a token-led, theme-aware Web Component system. The generated CSS files remain the source of truth:

- `src/tokens/css/mui-brand.css` defines primitive brand tokens generated by Style Dictionary.
- `public/css/mui-tokens.css` maps primitives into semantic, theme, and component tokens.
- `public/css/mui-reset.css` applies base page defaults.
- `public/css/mui-base.css` applies baseline native element styling.
- `src/muibook/css/app.css` defines Muibook app shell styling and the Muibook yellow brand scale.

This document is a human-readable design guide for agents, plugins, and contributors. Do not treat it as a replacement for token CSS, component source, `api.ts`, `doc.ts`, or the generated CEM.

## Token Source Boundaries

Keep token layers separated:

- Brand tokens are primitives only: raw color scales, base size scales, and other brand-owned raw values.
- Semantic tokens translate primitives into product meaning: text, surface, border, outline, feedback, form, action, and overlay.
- Component tokens translate semantic decisions into component behavior and styling.

Do not add semantic, theme, or component tokens to the brand token JSON/source. If a token name describes usage or intent, such as `surface`, `feedback`, `action`, `form`, `media-player`, `drawer`, or `select`, it belongs outside the brand primitive layer.

When updating generated token CSS, change the correct token source and run the token build. Do not patch generated token output by hand.

## Design Direction

Muibook is built around restrained, practical interface patterns:

- Framework-agnostic Web Components with native browser behavior where it remains reliable.
- Composable slots instead of heavy one-off component APIs.
- Light and dark mode support through semantic tokens, not duplicated component logic.
- Dense, scannable operational UI with clear hierarchy, stable spacing, and familiar controls.
- Progressive enhancement for richer surfaces, such as media controls and customizable select.
- CSS-first layout behavior where possible, with JS reserved for state, measurement, or browser APIs.

## Theme Model

Muibook uses explicit theme scopes:

- `html[data-theme="light"]`
- `html[data-theme="dark"]`

Light mode is the clean default surface direction. Dark mode reverses the same semantic system onto dark neutrals. Components should consume semantic and component tokens first, not raw primitive colors, unless a visual must remain locked across modes.

Use light and dark tokens through:

- Text: `--text-color`, `--text-color-optional`, `--text-color-success`, `--text-color-warning`, `--text-color-error`, `--text-color-info`
- Surface: `--surface`, `--surface-elevated-*`, `--surface-recessed-*`
- Border and focus: `--border-color`, `--outline-color`
- Feedback: `--feedback-*-border-color`, `--feedback-*-background`, `--feedback-*-icon`, `--feedback-*-text`
- Action: `--action-*-background`, `--action-*-border-color`, `--action-*-text-color`

## Surface System

Light mode maps surfaces to white and light greys:

| Token | Light mode direction |
| --- | --- |
| `--surface` | Base page surface |
| `--surface-elevated-100` | White elevated surface |
| `--surface-elevated-200` | Soft raised grey |
| `--surface-elevated-300` | Stronger raised grey |
| `--surface-recessed-100` | Subtle recessed grey |
| `--surface-recessed-200` | Stronger recessed grey |

Dark mode maps the same aliases to dark greys:

| Token | Dark mode direction |
| --- | --- |
| `--surface` | Main dark page surface |
| `--surface-elevated-100` | Elevated dark surface |
| `--surface-elevated-200` | Slightly deeper elevated dark surface |
| `--surface-elevated-300` | Deep elevated dark surface |
| `--surface-recessed-100` | Recessed dark surface |
| `--surface-recessed-200` | Deeper recessed dark surface |
| `--backdrop-overlay` | Modal and drawer overlay scrim |

Prefer these aliases over raw `--grey-*` values in component CSS.

## Typography

Base typography uses system fonts:

```css
--font-family: system-ui, sans-serif, "Apple Color Emoji", "Segoe UI Emoji";
```

The root scale uses:

```css
--root-font-size: 62.5%;
```

That makes `1rem` roughly `10px`, so `1.6rem` maps to `16px`.

| `--font-size-25` | `1.2rem` | 12px |
| `--font-size-50` | `1.4rem` | 14px |
| `--font-size-100` | `1.6rem` | 16px |
| `--font-size-200` | `1.8rem` | 18px |
| `--font-size-300` | `2.1rem` | 21px |
| `--font-size-400` | `2.4rem` | 24px |
| `--font-size-500` | `3.6rem` | 36px |

Text aliases:

| Token | Maps to |
| --- | --- |

Use `font-variant-numeric: tabular-nums` for time, counters, and changing numeric values.

| Token | Value | Approx px |
| --- | ---: | ---: |
| `--space-000` | `0` | 0px |
| `--space-025` | `0.2rem` | 2px |
| `--space-050` | `0.4rem` | 4px |
| `--space-100` | `0.6rem` | 6px |
| `--space-200` | `0.8rem` | 8px |
| `--space-300` | `1.2rem` | 12px |
| `--space-400` | `1.6rem` | 16px |
| `--space-500` | `2.4rem` | 24px |
| `--space-600` | `3.6rem` | 36px |
| `--space-700` | `4.8rem` | 48px |
| `--space-800` | `6rem` | 60px |

Do not invent intermediate space tokens in component CSS. If a hard value matches a spacing token, use the token.

## Radius, Border, Shadow

Radius should stay practical and restrained:

| Token | Value |
| --- | ---: |
| `--radius-000` | `0` |
| `--radius-100` | `4px` |
| `--radius-150` | `6px` |
| `--radius-200` | `8px` |
| `--radius-300` | `16px` |
| `--radius-400` | `24px` |
| `--radius-500` | `36px` |
| `--radius-600` | `48px` |

Use semantic border and shadow tokens:

- `--border-thin`, `--border-medium`, `--border-thick`
- `--outline-thin`, `--outline-medium`, `--outline-thick`
- `--shadow-thin`, `--shadow-medium`, `--shadow-thick`

Do not add component-specific escape hatches unless the component has a real public customization need.

## Action Language

Buttons and links share the action system:

| Variant | Use |
| --- | --- |
| `primary` | Main action |
| `secondary` | Clear alternate action |
| `tertiary` | Low-emphasis action |
| `overlay` | Action over image/video/dark overlay surfaces |
| `attention` | Destructive or urgent action |

Action sizing uses shared tokens:

| Size | Typical use |
| --- | --- |
| `xx-small` | Very dense controls and compact metadata |
| `x-small` | Compact controls |
| `small` | Secondary controls |
| `medium` | Default actions |
| `large` | Prominent actions |

Use `pending` for async interaction blocking when the action should not visually become disabled.

## Feedback Language

Feedback tokens support neutral, positive, info, warning, and attention states:

| Purpose | Example use |
| --- | --- |
| Neutral | Passive status, default message, non-urgent badge |
| Positive | Success, completed, valid |
| Info | Informational state or system guidance |
| Warning | Caution, incomplete, needs review |
| Attention | Error, destructive, urgent |

Use feedback tokens for color intent. Avoid raw red/orange/green/blue values in component CSS.

## Forms

Form components should use:

- `--form-default-*`
- `--form-success-*`
- `--form-warning-*`
- `--form-error-*`

Label association, error messaging, optional state, and native disabled semantics should remain accessible. Rich/custom form experiments should preserve native semantics where possible.

## Media And Overlay Surfaces

Media Player has two visual modes:

- Plain audio/control surfaces should respond to light and dark mode.
- Video, image artwork, and image-backed audio overlays should lock to dark overlay behavior for readable controls.

Use overlay tokens for media overlays:

- `--media-player-dark-control-color`
- `--media-player-dark-overlay-background`
- `--media-player-dark-overlay-controls-background`
- `--media-player-center-action-overlay-background`
- `--media-player-center-action-overlay-background-hover`
- `--action-overlay-*`

Use regular surface tokens for non-image audio states.

## Composition

Prefer Muibook layout components over ad hoc local CSS:

- `mui-v-stack` for vertical rhythm.
- `mui-h-stack` for inline composition.
- `mui-grid` for repeatable layout tracks.
- `mui-responsive` for viewport or container switching.
- `mui-card`, `mui-slat`, and `mui-list` for structured content groups.

Slots should carry composition. Components should not over-own authored content unless the component is providing a clear native behavior or accessibility contract.

## Responsive Behavior

Use CSS and `mui-responsive` before bespoke JS:

- Use viewport responsive behavior for page-level changes.
- Use `variant="container"` when a component needs to respond to its containing layout area.
- Keep mobile controls stable, avoid overlap, and prefer wrapping before shrinking text.

## Icons And Illustrations

Icons are direct, functional controls. Prefer existing Muibook icons over custom SVG in component stories.

Illustrations are inspectable visual accents with their own size tokens:

| Token | Use |
| --- | --- |
| `--illustration-size-x-small` | Dense rich-option or compact tile |
| `--illustration-size-small` | Small visual support |
| `--illustration-size-medium` | Default visual support |
| `--illustration-size-large` | Large visual support |
| `--illustration-size-x-large` | Feature visual support |

Use `color` on illustrations when an example needs semantic color, preferably with feedback tokens.

## Documentation And Knowledge

The knowledge output is split by concern:

- `custom-elements.json` contains CEM-generated component API and docs guidance.
- `dynamic-attrs.json` contains runtime/destination structural attributes.
- `rules.ts` contains global generation rules.
- `compositions.ts` contains composition examples and curated agent subsets.
- `keywords.ts` contains keyword mappings and curated agent subsets.
- `AGENTS.md` contains agent operating guidance.
- `DESIGN.md` contains this design-language overview.

Keep `AGENTS.md` and `DESIGN.md` at the repository root so agent tooling can discover them. The export script copies them into `muibook-knowledge`.

## Maintenance Notes

- Update tokens first when changing visual language.
- Keep component docs in `api.ts` and `doc.ts` so CEM remains the API/UX source.
- Keep dynamic attrs separate from API props.
- Keep `src/knowledge` focused on importable rules, keywords, and compositions.
- Keep this document aligned with the generated CSS, but do not duplicate every token.
